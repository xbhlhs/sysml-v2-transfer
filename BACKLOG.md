# 项目路线图

这是仓库内的本地规划文件，不会同步到 GitHub issue 视图。

## 版本管理原则

- 项目按语义版本管理。
- `0.x` 表示产品仍在快速演进，接口和工作流可能继续调整。
- 当最小可用闭环完成时，才考虑切换到 `1.0.0`，在此之前保持 `0.x` 分期。
- 版本号应反映用户可感知能力的变化，而不是单纯的内部重构。
- 建议规则：
  - `patch` 用于修复缺陷和纯文档更新
  - `minor` 用于增加新的可见能力
  - `major` 用于改变核心契约或工作流

## 分期目标

### v0.1.0 - CLI 先出可视化结果
目标：先让命令行能输出可直接打开查看的可视化文件。

准入条件：
- 已有稳定的最小文本输入路径
- 已定义 CLI 输出路径和错误处理约定
- 已有基础转换链路的实现起点

准出条件：
- CLI 可以生成可直接打开的 SVG / canvas 类可视化结果
- 支持文件输入与输出路径
- 转换失败时返回清晰错误信息
- 具备最小的可视化回归测试

完成什么：
- 命令行可以把 SysML v2 输入转成一个本地可打开的可视化文件
- 用户可以指定输入文件和输出文件
- 输出结果能够被本地工具直接查看
- 基础错误路径可定位

不做什么：
- 不追求完整 SysML v2 语义覆盖
- 不做复杂布局编辑
- 不做 GUI 交互能力扩展
- 不做反向回写闭环

风险点：
- 解析范围过小会限制可视化结果的可用性
- 输出格式选择不当会影响后续 GUI 复用
- CLI 参数设计过早定死可能影响后续扩展

关联 issue：

### v0.2.0 - GUI 复现可视化流程
目标：让 GUI 能完成与 CLI 相同的可视化流程。

准入条件：
- CLI 已经能够稳定导出可视化结果
- 可视化能力已经以稳定接口方式暴露
- GUI 能够消费现有转换结果

准出条件：
- GUI 可以加载 SysML v2 输入
- GUI 可以触发可视化生成
- GUI 可以预览并导出可视化结果
- GUI 与 CLI 的输出结果保持一致性

完成什么：
- GUI 可以完成 CLI 同样的可视化流程
- GUI 可以直接预览生成结果
- GUI 可以导出可视化文件
- GUI 不依赖额外手动切换到命令行

不做什么：
- 不做复杂多视图切换
- 不做属性编辑器
- 不做图像处理工具集
- 不做高级布局编辑

风险点：
- GUI 与 CLI 的结果一致性需要严格约束
- GUI 依赖链过重会降低启动和调试效率
- 如果核心转换接口不稳定，GUI 会被动跟着抖动

关联 issue：
- #13
- #11
- #6

### v1.0.0 (Apollo) - Definition View 闭环
目标：通过 CLI 从 texture notation 生成 Definition View，形成第一个可用闭环。

准入条件：
- 已完成 CLI 的稳定可视化路径
- 已明确 Definition View 的投影规则
- 已建立事件驱动解析与轻量 AST 链路

准出条件：
- CLI 可以稳定输出 `*-def-view.svg`
- Definition View 的 package / definition / usage 投影规则清晰
- 典型 `.sysml` 样例可以通过测试验证

完成什么：
- 通过 CLI 从 texture notation 生成 Definition View
- 支持当前 `pla.sysml` 这类结构完整样例
- 输出结果可直接打开查看
- 同一份数据为后续视图复用奠定基础

不做什么：
- 不做其他视图的完整实现
- 不做反向回写闭环
- 不做 GUI 深度交互扩展
- 不把 package 作为 Definition View 的主视觉节点

风险点：
- Definition View 规则如果过早固化，会影响后续视图复用
- 过度贴近 v1 会削弱 v2 原生视图语义
- 输出布局如果不稳定，会影响后续图形视图扩展

关联 issue：
- #12
- #20

### v1.2.x / v2.x.x - 后续视图投影
目标：在同一份解析结果和中间结构上，继续投影其他 SysML v2 视图。

准入条件：
- `Definition View` 已稳定
- 统一的解析结果和中间结构已可复用
- 视图投影边界明确

准出条件：
- 至少一个后续视图可以稳定生成 SVG
- 不同视图共用同一份解析结果
- 视图之间可以切换或并行生成

完成什么：
- Usage View
- Interconnection View
- Action Flow View
- Interaction View
- State View
- Requirement View

不做什么：
- 不把后续视图塞回 `1.0.0` 的闭环范围
- 不要求所有视图同时完成
- 不要求一次性覆盖全部 SysML v2 语义

风险点：
- 视图过多会拉高维护成本
- 共用解析层一旦变化，会影响多个视图
- 不同视图的抽象边界需要持续收敛

关联 issue：
- #21
- #22
- #23
- #24
- #25
- #26

### v1.1.x - GUI 能力扩展
目标：在预览和导出之外，扩展 GUI 的交互能力。

准入条件：
- GUI 已具备基础可视化与导出能力
- 视图和转换链路足够稳定，能够承载更多交互
- 反向回写能力至少达到最小闭环水平

准出条件：
- GUI 支持多视图切换
- GUI 支持对象属性 / 元数据编辑
- GUI 支持图像或可视化结果的基础操作
- 新交互不会破坏已有可视化工作流

完成什么：
- GUI 从“查看和导出”扩展到“多视图和轻量编辑”
- 用户可以在 GUI 中补充对象信息
- 用户可以对可视化结果做基础处理

不做什么：
- 不做完整建模器
- 不做复杂布局编辑器
- 不做重型图像编辑工作流
- 不做新的核心转换路径

风险点：
- GUI 交互膨胀会拖慢稳定性
- 编辑能力一旦出现，保存和回写链路会变复杂
- 图像操作如果过重，会偏离 SysML 工具主线

关联 issue：
- #14
- #15
- #16
- #11
